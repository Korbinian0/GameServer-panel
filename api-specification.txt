API-SPECIFICATION FOR GATEWAY <-> BACKEND NODES

ÜBERSICHT
=========
Diese Spezifikation beschreibt die gesamte gRPC-Schnittstelle zwischen dem zentralen Gateway und den Backend-Nodes (Linux und Windows).

Alle Backend-Komponenten müssen sich exakt an diese Spezifikation halten.

Authentifizierung, Berechtigungen, Fehlercodes und Events sind Teil der offiziellen API.

Allgemeine Prinzipien
---------------------
- Alle Methoden sind gRPC-RPCs.
- Nachrichten werden als Protobuf-Modelle verstanden, dokumentiert in JSON-ähnlicher Form.
- Streaming-Endpunkte sind explizit markiert.
- Alle Requests müssen gültige Felder enthalten, sonst muss ein Fehler mit dem passenden Fehlercode zurückgegeben werden.
- Authentifizierung erfolgt mit Backend-Token + TLS. Optional kann mTLS für zusätzliche Sicherheit erzwungen werden.

AUTHENTIFIZIERUNGSVERFAHREN
===========================
Backend-Token + TLS
-------------------
- Backend-Nodes erhalten einen eindeutigen, vom Gateway ausgestellten Backend-Token.
- Jeder Aufruf muss den Token in den gRPC-Metadaten enthalten:
  - Header-Key: `authorization`
  - Header-Wert: `Bearer <backend-token>`
- TLS muss auf allen Verbindungen aktiviert sein.
- Optional: mTLS, wenn Gateway und Backend sich gegenseitig authentifizieren müssen.

Token-Validierung
------------------
- Der Backend-Token muss beim Gateway validiert werden.
- Abgelaufene oder entziehene Tokens müssen mit Fehlercode `UNAUTHENTICATED` / `AUTH_1001` beantwortet werden.

Berechtigungen
===============
- Jede Methode verlangt eine spezifische Berechtigung.
- Berechtigungen werden als String angegeben, z.B. `server.start`.
- Das Gateway prüft Berechtigungen vor der Ausführung.

Fehlercodes
===========
Standardisierte Fehlercodes für Backend-Implementierung:

Allgemein
--------
- `OK` (0): Erfolg.
- `INVALID_ARGUMENT` (3) / `ARG_1000`: Ungültige Eingabe.
- `NOT_FOUND` (5) / `RES_1001`: Ressource nicht gefunden.
- `ALREADY_EXISTS` (6) / `RES_1002`: Ressource existiert bereits.
- `PERMISSION_DENIED` (7) / `AUTH_1000`: Fehlende Berechtigung.
- `UNAUTHENTICATED` (16) / `AUTH_1001`: Ungültiger oder fehlender Token.
- `UNAVAILABLE` (14) / `SYS_1000`: Node nicht erreichbar oder temporär nicht verfügbar.
- `INTERNAL` (13) / `SYS_1001`: Interner Fehler im Backend-Node.
- `RESOURCE_EXHAUSTED` (8) / `SYS_1002`: Ressourcenlimit erreicht.
- `FAILED_PRECONDITION` (9) / `SYS_1003`: Zustand nicht geeignet für Aktion.
- `DEADLINE_EXCEEDED` (4) / `SYS_1004`: Zeitlimit überschritten.
- `UNKNOWN` (2) / `SYS_1999`: Unbekannter Fehler.

Spezielle Fehlercodes
---------------------
- `AUTH_1002`: Token-Format ungültig.
- `AUTH_1003`: TLS-/mTLS-Validierung fehlgeschlagen.
- `PERM_1000`: Fehlende Methode-Berechtigung.
- `FILE_1000`: Dateioperation fehlgeschlagen.
- `SERVER_1000`: Serverprozess-Operation fehlgeschlagen.
- `EVENT_1000`: Event-Verarbeitung fehlgeschlagen.
- `STREAM_1000`: Streaming-Verbindung fehlgeschlagen.

Events
======
Event-Typen werden vom Backend aktiv an das Gateway gesendet und sind in der Protobuf-Schnittstelle definiert.

Verfügbare Event-Typen
----------------------
- `BackendStatusUpdated`
- `ServerStateChanged`
- `ConsoleOutput`
- `FileTransferProgress`
- `HealthReport`
- `SecurityAlert`
- `NodeLogEntry`
- `PermissionChanged`

Jedes Event besitzt einen `eventType` und einen `payload`.

SERVICE: NodeManagement
=======================

METHOD: RegisterNode
--------------------
NAME: RegisterNode
BESCHREIBUNG:
Der Backend-Node registriert sich beim Gateway. Diese Methode wird einmalig bei Start oder nach Neustart aufgerufen.
REQUEST:
{
  "nodeId": "string",
  "platform": "Linux|Windows",
  "hostname": "string",
  "ipAddress": "string",
  "capabilities": ["string"],
  "version": "string"
}
RESPONSE:
{
  "success": true,
  "assignedNodeId": "string",
  "expiresAt": "string (ISO 8601)",
  "message": "string"
}
AUTHENTIFIZIERUNG:
Backend Token + TLS
BERECHTIGUNG:
node.register
BEISPIEL:
REQUEST:
{
  "nodeId": "node-123",
  "platform": "Linux",
  "hostname": "game-node-01",
  "ipAddress": "10.0.0.100",
  "capabilities": ["server.start", "file.upload", "console.stream"],
  "version": "1.0.0"
}
RESPONSE:
{
  "success": true,
  "assignedNodeId": "node-123",
  "expiresAt": "2026-12-31T23:59:59Z",
  "message": "Node registered successfully"
}

SERVICE: Console
================

METHOD: OpenConsoleStream
-------------------------
NAME: OpenConsoleStream
BESCHREIBUNG:
Öffnet einen bidirektionalen Konsolenstream zwischen Gateway und Backend-Serverprozess.
REQUEST:
ConsoleOpenRequest
{
  "nodeId": "string",
  "serverId": "string",
  "sessionId": "string"
}
RESPONSE:
Bidirectional Stream
- `ConsoleRequest` vom Gateway an Backend:
  - `sessionId`: string
  - `command`: string
  - `resizeWidth`: int (optional)
  - `resizeHeight`: int (optional)
- `ConsoleResponse` vom Backend an Gateway:
  - `sessionId`: string
  - `output`: string
  - `isError`: bool
  - `timestamp`: string
AUTHENTIFIZIERUNG:
Backend Token + TLS
BERECHTIGUNG:
console.access
BEISPIEL:
REQUEST:
{
  "nodeId": "node-123",
  "serverId": "srv-001",
  "sessionId": "sess-789"
}
STREAM:
Gateway -> Backend:
{
  "sessionId": "sess-789",
  "command": "status"
}
Backend -> Gateway:
{
  "sessionId": "sess-789",
  "output": "Server is running\n",
  "isError": false,
  "timestamp": "2026-06-05T12:01:00Z"
}

STREAMING-ENDPUNKTE
===================
- `UploadFile`: Client-streaming von Gateway zu Backend.
- `DownloadFile`: Server-streaming von Backend zu Gateway.
- `OpenConsoleStream`: Bidirektionales Streaming für Konsoleninteraktion.
- `StreamServerLogs`: Server-streaming von Backend zu Gateway.
- `StreamNodeEvents`: Server-streaming von Backend zu Gateway.

IMPLEMENTIERUNGS-ANFORDERUNGEN
=============================
- Backend-Node muss TLS unterstützen.
- Backend-Node muss Backend-Token verifizieren und fehlgeschlagene Authentifizierung korrekt melden.
- Alle Methoden müssen klar benannte Response-Objekte liefern.
- Events müssen konsistent gesendet werden und den angegebenen Payload-Typ verwenden.
- Node-spezifische Aktionen müssen den `nodeId` als identifizierenden Kontext akzeptieren.
- Methoden mit Streaming müssen den Zustand korrekt managen und `isLastChunk` verwenden.

HINWEIS ZUR VERSIONIERUNG
=========================
- Diese Spezifikation beschreibt die aktuelle API-Version 1.
- Erweiterungen müssen rückwärtskompatibel sein und neue Felder optional machen.
- Änderungen an Service- oder Methodennamen sind breaking changes und dürfen nur nach Abstimmung erfolgen.
